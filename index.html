Task № 1

Please write a function compile_csv_search(text, key_name) that parses text in the CSV format.

( You are not required to handle quoting and escaping in values; assume field values never contain commas or other special characters.)

Your function must return a function that looks up a record by a value of the field specified as the second argument to compile_csv_search. Assume that all values in the key field are unique.

Sample usage:


var csv_by_name = compile_csv_search(
    "ip,name,desc\n"+
    "10.49.1.4,server1,Main Server\n"+
    "10.52.5.1,server2,Backup Server\n",
    "name");
console.log(csv_by_name("server2"));// >>> {ip: "10.52.5.1", name: "server2", desc: "Backup Server"}
console.log(csv_by_name("server9")); // >>> undefined

Task № 2

You have the voluntary array of values. Please, write a function that will find all integer duplicates and returns their qty and its number.

  • If array contains several duplicates it should return the most frequent value.
  • If array has the same length of duplicates it should return the biggest number

Example:


[0, 1, 2, 3, 3, 5, 7]    // ->>> Qty: 2, number: 3
[0, 1, 1, 2, 2, 4]        // ->> Qty: 2, number: 2
[0, 1, 1, 1,  2, 2, 4]   // ->> Qty: 3, number: 1

Part 2

  • Do not use any 3rd party libraries, frameworks, just pure JS and CSS.
  • Each function and method should be commented.
  • Use NODE.js for http server

You have Design Mock Up (see PDF) and you are responsible for UI implementation. Example of test data is represented into JSON file. This data is required to build a table:

First table (Sync Journal):

For each syncDate you should show:

  • a) Qty of JSON objects in column - sync elements;
  • b) State if at least one status == ERROR return Failed , if OK >> ok,
  • с) Сonflicts - number of elements with status == Conflict

Second table (contains details of synchronization ‘Sync Details’):

  • If user clicks on the row in the first table the second one shows IDs that grouped by Type.

The design has ‘Start Sync’ button and click on the button should send http request to load new data into the first table. Animation of request/loader should be synchronized with request (data should be added to the table if animation finished).

For ‘Sync Mode’ select: if ‘Scheduled’ selected user should see ‘Schedule’ table (table is just a mock). Also, the test app should have custom ‘back’ and ‘next’ button to appropriate screen.

Architectural aspect:

Your application should follow to the next approach:

1) Global route - app.js that will be responsible for managing js modules.

It also includes a module deletion and initialization. You can load a module, for example, by ID, that is wired up with some element ( in our case it’s ‘synchronize’ - see Design Mock Up pdf ) and this ID is used into SCREEN_MANAGER.js like a key of object. If a user opens some page, the app.js loads a module and initializes it. Initialized module starts load required template and calls .render() method. The app.js should be available for all pages. If user leaves from particular page the route should destroy attached modules.

2) Page module - a class that is responsible for logic of the whole page (rendering, calculations etc).

3) “SCREEN_MANAGER.js” contains path information about your js/html/css. This is plain object where a ‘key’ is module ID. This object should be available for all pages like app.js.

4) Implement error handling. If your request is failed or something was not initialized then you should return a user to previous page and to show error message. In such case, the app.js should delete called module by calling .destroy() method. In case of other errors, just show error message.

Your html should be divided on the next partials:

  • grid.html (grid of icons);
  • schedule.htm (schedule table);
  • syncDetails.html (Sync Details table)
